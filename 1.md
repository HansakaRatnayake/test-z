## Test Design & Implementation
### Test Design
The activity of deriving and specifying test cases from test conditions.
- **Test Case:** Preconditions, inputs, actions, expected results, and postconditions.
- **Test Data:** Data created to satisfy execution preconditions and inputs.
- **Key Activities:** Prioritizing cases, identifying data, and designing the environment.

### Test Implementation
The activity that prepares the testware needed for execution.
- **Major Activities:** Developing test procedures, creating automated scripts, and arranging test suites into an execution schedule.

---

## Test Execution & Completion
### Test Execution
The process of running tests on a component or system and producing actual results.
- **Anomalies:** Discrepancies between actual and expected results must be analyzed to identify defects or false positives.

### Test Completion
Activities that make test assets available for later use and communicate results to stakeholders.
- **Milestones:** Occurs at software release, project completion, or end of an Agile iteration.
- **Key Activities:** Closing defect reports, creating a **Test Summary Report**, and performing retrospectives.

---

## Test Work Products
Generic name for any documentation or artifact used in testing:
- **Planning:** Test Plans.
- **Monitoring:** Test Reports.
- **Analysis:** Test Conditions.
- **Design:** Test Cases.
- **Implementation:** Test procedures, suites, and schedules.
- **Execution:** Status logs and defect reports.
- **Completion:** Test summary reports and improvement action items.

---

## Traceability
The relationship between two or more work products.
- **Bi-directional Traceability:** The ability to trace from requirements to test results and vice-versa.
- **Benefits:** Impact analysis, auditability, IT governance compliance, and assessing product quality against business goals.

## Software Development Life Cycle (SDLC) & Testing
The chosen life cycle impacts how testing is organized:
- **Sequential (Waterfall):** Testing typically occurs at the end. Difficult to provide early feedback.
- **V-Model:** Every development phase has a corresponding testing phase. Encourages early testing.
- **Iterative/Agile:** Testing is integrated throughout the process (Scrum, Kanban, XP).

## Testing Types
### White-Box Testing
Testing based on internal structure (code is visible).
- **Techniques:** Statement Coverage, Decision Coverage, Branch Coverage, Condition Coverage, and Path Coverage.

### Black-Box Testing
Behavioral testing based on requirements (code is not visible).
- **Focus:** Inputs and outputs. Includes functional and regression testing.

### Gray-Box Testing
Testing with partial knowledge of the internal code structure.

---

## Non-Functional & Change-Related Testing
### Non-Functional Testing
Tests "how well" the system works.
- **Performance:** Speed and stability.
- **Load/Stress:** Capacity and robustness under heavy conditions.
- **Usability:** User-friendliness.
- **Security:** Vulnerability assessment.

### Change-Related Testing
- **Confirmation Testing (Re-testing):** Verifying that a specific defect has been fixed.
- **Regression Testing:** Ensuring new changes haven't negatively impacted existing functionality.
